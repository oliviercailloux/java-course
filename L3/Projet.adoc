== Généralités
- Deux superviseurices auront connaissance de l’état actuel de la branche dev.
- Sur volontariat, le projet sera éventuellement livré en open source en fin d’année.

== Que livrer
- Si branche dev diffère de branche main, votre principale responsabilité est de restaurer la fonctionnalité main
- Retirer les tâches faites du TODO, ajouter des propositions éventuellement
- Documenter en https://docs.asciidoctor.org/asciidoc/latest/[Asciidoc] et en https://plantuml.com/[PlantUML] dans le répertoire `Doc`, d’une façon accessible depuis le README.
- Minimisez les changements : ne changez pas les espaces en fin de ligne, ne reformatez pas le code existant…
- Formattez correctement _votre_ code !

== Comment livrer
- Livrer sur branche dev. Fusionner sur branche main si pas de régression
- Un étudiant non superviseurice ne peut pousser sur les branches dev et main que le jour de son créneau
- Remettre sur Moodle : indiquer pour chaque tâche un lien vers la https://docs.github.com/en/pull-requests/committing-changes-to-your-project/viewing-and-comparing-commits/comparing-commits\#comparing-commits[diff] entre le commit de la position précédente de dev et sa nouvelle position (exemple : si deux tâches accomplies, indiquer deux liens, un de C1 à C3 et un de C3 à C7, où C1 est l’ancien commit de dev, C3 le commit après la première tâche, C7 le commit après la deuxième tâche et le nouveau commit de dev) ; indiquer si fusionné sur main ou pourquoi pas
- au plus tard à la fin du jour de votre créneau.
